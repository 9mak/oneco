# Implementation Plan

## Tasks

- [x] 1. データベースモデルとマイグレーション
- [x] 1.1 (P) ユーザー管理テーブルの作成
  - Users テーブル: 暗号化LINEユーザーID、アクティブフラグ、タイムスタンプ
  - アクティブユーザー検索用の部分インデックス作成
  - 暗号化フィールドには十分な長さ（255文字）を確保
  - _Requirements: 1.1, 7.1_

- [x] 1.2 (P) 通知条件テーブルの作成
  - NotificationPreferences テーブル: ユーザー外部キー、種別、都道府県（JSON配列）、年齢範囲、サイズ、性別、通知有効フラグ
  - ユーザーとの1:0..1関係を一意制約で保証
  - アクティブな通知条件検索用の部分インデックス作成
  - _Requirements: 1.3, 1.4_

- [x] 1.3 (P) 通知履歴テーブルの作成
  - NotificationHistory テーブル: ユーザー外部キー、動物ソースURL、ステータス、通知日時
  - ユーザーIDと動物URLの組み合わせに一意制約（重複通知防止）
  - 履歴検索用の複合インデックスと日付インデックス作成
  - カスケード削除の設定
  - _Requirements: 5.1, 5.2, 5.3_

- [x] 2. ドメインモデルと暗号化機能
- [x] 2.1 ユーザードメインモデルの実装
  - ユーザーエンティティ: 識別子、暗号化LINEユーザーID、アクティブ状態
  - 通知条件エンティティ: 種別、都道府県リスト、年齢範囲、サイズ、性別、有効フラグ
  - マッチング結果データ: ユーザーID、暗号化LINEユーザーID、条件ID、マッチスコア
  - _Requirements: 1.1, 1.3_

- [x] 2.2 暗号化/復号ユーティリティの実装
  - Fernet暗号化（AES-128 + HMAC）によるLINEユーザーID暗号化
  - 環境変数からの暗号化キー取得
  - 暗号化・復号の双方向処理
  - _Requirements: 7.1_

- [x] 3. リポジトリ層
- [x] 3.1 ユーザーリポジトリの実装
  - ユーザーの新規登録（暗号化LINEユーザーIDで検索・挿入）
  - 暗号化LINEユーザーIDによるユーザー検索
  - ユーザー無効化（アクティブフラグ更新）
  - アクティブユーザー一覧取得
  - _Requirements: 1.1, 1.4, 1.6, 7.5_

- [x] 3.2 通知条件リポジトリの実装
  - 条件の作成・更新・取得
  - ユーザーIDによる条件検索
  - 通知有効フラグの切り替え
  - アクティブな通知条件の一括取得（マッチング用）
  - _Requirements: 1.3, 1.4, 1.5, 1.6, 1.7, 3.1_

- [x] 3.3 通知履歴リポジトリの実装
  - 通知履歴の記録（ユーザーID、動物URL、ステータス）
  - ユーザーと動物の組み合わせによる重複チェック
  - 90日超過した履歴の自動削除/アーカイブ処理
  - _Requirements: 5.1, 5.2, 5.3, 5.4, 5.5_

- [x] 4. LINEアダプター層
- [x] 4.1 LINE Messaging API連携の実装
  - line-bot-sdk を使用したプッシュメッセージ送信
  - チャネルアクセストークンの管理
  - Webhook署名検証機能
  - _Requirements: 4.1, 7.4_

- [x] 4.2 エラーハンドリングとリトライ機能
  - LINE API エラー（401、429、5xx）の検出と処理
  - 指数バックオフによるリトライ（最大3回）
  - レート制限（retry_after）への対応
  - 送信結果の構造化データ返却
  - _Requirements: 4.3, 4.4, 4.5, 4.6, 6.3_

- [x] 4.3 通知メッセージフォーマットの実装
  - 動物情報（種別、性別、年齢、サイズ、地域、URL）を含むメッセージ構築
  - 人間が読みやすいフォーマット
  - _Requirements: 4.2_

- [x] 5. ドメインサービス層
- [x] 5.1 ユーザーサービスの実装
  - 友だち追加時のユーザー新規登録（暗号化処理含む）
  - 通知条件の更新処理
  - 通知の有効/無効切り替え
  - ブロック/削除時のユーザー無効化
  - _Requirements: 1.1, 1.4, 1.5, 1.6, 1.7, 7.5_

- [x] 5.2 マッチングサービスの実装
  - アクティブな通知条件の一括取得
  - 動物データと各条件の照合（種別、都道府県、年齢、サイズ、性別）
  - AND条件による完全マッチ判定
  - 早期リターンによる効率化（最初の不一致で即座にスキップ）
  - 条件未設定項目は「すべて許可」として処理
  - マッチング結果のログ記録
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 3.5_

- [x] 5.3 通知サービスの実装
  - 新着動物データの受信と処理オーケストレーション
  - マッチング → 重複チェック → 配信の一連フロー制御
  - 非同期処理による並列送信（asyncio.gather、最大10並列）
  - バッチ処理（100件/バッチ）
  - 送信結果の記録と処理結果サマリー返却
  - _Requirements: 4.1, 8.1, 8.2, 8.3, 8.4, 8.5_

- [x] 5.4 送信失敗時のリトライと運用者通知
  - リトライ後も失敗した場合の運用者通知（既存Slack連携活用）
  - 失敗記録の保存と処理継続
  - _Requirements: 4.5, 6.2_

- [x] 6. API層（Webhookコントローラー）
- [x] 6.1 data-collector Webhookエンドポイントの実装
  - POST /api/v1/notifications/webhook エンドポイント
  - APIキー認証（X-API-Keyヘッダー検証）
  - 新着動物データのバリデーション（AnimalData形式）
  - HTTP 202 Accepted の即時返却
  - バックグラウンドタスクでの通知処理起動
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5, 7.2, 7.3_

- [x] 6.2 LINE Webhookエンドポイントの実装
  - POST /api/v1/line/webhook エンドポイント
  - LINE署名検証（X-Line-Signature）
  - Followイベント（友だち追加）の処理
  - Unfollowイベント（ブロック/削除）の処理
  - メッセージイベントの受信（条件設定コマンド用の基盤）
  - _Requirements: 1.1, 7.4, 7.5_

- [x] 6.3 ヘルスチェックエンドポイントの実装
  - GET /health エンドポイント
  - データベース接続状態の確認
  - LINE API接続状態の確認（オプショナル）
  - サービス状態（healthy/degraded/unhealthy）の返却
  - _Requirements: 6.6_

- [x] 7. 条件設定対話機能
- [x] 7.1 コマンド解析と対話フロー制御
  - 設定コマンドの認識（例: 「設定」「条件変更」「停止」「再開」）
  - 対話状態の管理（設定フロー中のステップ追跡）
  - 各条件項目の順次入力フロー
  - _Requirements: 1.2, 1.5, 1.6, 1.7_

- [x] 7.2 条件入力とLINEメッセージ送信
  - 条件入力プロンプトの送信（種別選択、都道府県選択など）
  - 入力値のバリデーション
  - 設定完了・変更完了・停止・再開の確認メッセージ送信
  - _Requirements: 1.2, 1.3, 1.4, 1.5, 1.6, 1.7_

- [x] 8. 監視とメトリクス
- [x] 8.1 メトリクス記録の実装
  - 1時間あたりの通知送信数の記録
  - API応答時間（p50、p95、p99）の記録
  - エラー発生率の記録
  - マッチング処理時間の記録
  - _Requirements: 6.4_

- [x] 8.2 アラート機能の実装
  - エラー率閾値（5%で警告、20%で緊急）によるアラート
  - API応答時間閾値（p95 > 5秒）によるアラート
  - 運用者への通知送信（既存Slack連携活用）
  - _Requirements: 6.5_

- [x] 8.3 エラーログと監査ログの実装
  - 認証エラー（無効APIキー、署名検証失敗）のログ記録
  - データベース接続エラーのログ記録と運用者通知
  - LINE API接続エラーのログ記録
  - 個人データアクセスの監査ログ
  - _Requirements: 6.1, 6.2, 6.3, 7.3, 7.6_

- [x] 9. 統合とエンドツーエンドテスト
- [x] 9.1 data-collector連携の統合テスト
  - Webhook受信 → マッチング → LINE送信の全フロー検証
  - APIキー認証の検証
  - 不正リクエストへのエラーレスポンス検証
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5, 3.1, 3.2, 3.3, 3.4, 3.5, 4.1_

- [x] 9.2 LINE連携の統合テスト
  - 友だち追加 → ユーザー登録の検証
  - 条件設定フローの検証
  - ブロック/削除 → 無効化の検証
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 1.6, 1.7, 7.5_

- [x] 9.3 重複通知防止と履歴管理のテスト
  - 同一動物への二重通知防止の検証
  - 履歴記録の正確性検証
  - 90日超過履歴の削除検証
  - _Requirements: 5.1, 5.2, 5.3, 5.4, 5.5_

- [x] 9.4 パフォーマンスとスケーラビリティのテスト
  - 大量ユーザーへのバッチ通知検証（100件/バッチ）
  - 並列送信（10並列）の動作検証
  - 平均応答時間5秒以内の検証
  - 接続プール枯渇時のキュー処理検証
  - _Requirements: 8.1, 8.2, 8.3, 8.4, 8.5_
