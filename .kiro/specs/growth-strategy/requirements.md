# oneco 集客戦略 (growth-strategy) 要件定義

## 1. 背景と前提

### 1.1 現在地（2026-06-12 時点）

| 項目 | 状態 |
|---|---|
| 公開状態 | https://frontend-psi-ten-73.vercel.app で稼働中（vercel.app デフォルトドメイン、HTTP 200） |
| データ | 1,200+ 件の保護動物データを Supabase に保持、日次自動収集 |
| 対応自治体 | 47 都道府県 91 ホスト |
| アクセス計測 | GA4 (`G-KEE66C1E8B`) 導入済み、本番計測中 |
| SEO 基盤 | Search Console プロパティ追加・所有権確認・sitemap.xml 送信済み |
| 独自ドメイン | **未取得**（`.com` で安く始めて `.jp` は後 案で保留中） |
| マネタイズ | 非営利・寄付ベース。コスト構造は月数百円規模（実質ゼロ運営） |

### 1.2 使命の再確認

oneco の使命は **「殺処分ゼロに近づける」** こと。直接的に殺処分を減らす経路は2つ:

1. **譲渡成立を増やす**: 里親候補が動物の存在を知り、自治体に問い合わせる導線を作る
2. **迷子の再会を増やす**: 飼い主が迷子情報を発見し、迎えに行く導線を作る

集客の優劣はこの 2 経路にどれだけ寄与するかで判断する。アクセス数自体は中間指標であり、本当の KPI は「自治体ページへの遷移」「再会成立報告」のような遠位指標。

### 1.3 解決したい問題

公開済みだがほぼ誰にも知られていない。SNS / プレス / 自治体・保護団体ネットワークへの露出がゼロで、Organic Search も sitemap 送信直後で未流入。**0 → 100 ユーザー/日** を最初に乗り越える必要がある。

### 1.4 制約

- **個人運営 (bus factor=1)**: 営業活動・コンテンツ制作にかけられる時間は週数時間
- **非営利・無料**: 広告費は数万円以下に抑える。基本はオーガニック施策
- **AI 自動化前提**: コンテンツ制作・SNS 投稿・反応分析は Claude / Codex で半自動化する
- **法的リスク**: 自治体の公開情報を再配信しているため、自治体との関係を毀損する施策は不可（[[project_legal_scraping]] 参照）
- **マネタイズ判断は保留**: 営利化前に弁護士相談が必要（[[project_monetization_legal_checkpoint]]）

## 2. 要件

### 要件 1: 計測の土台を完成させる

**Why**: 効果のある施策を続け、効かない施策を捨てる判断ができないと、限られた個人運営リソースが空転する。

#### 受け入れ基準

1.1. GA4 で以下のイベントが計測されている（自動 or 手動カスタム）:
   - `page_view`（自動・既存）
   - `external_link_click`（自治体公式サイト or 動物詳細元ページへの遷移＝**最重要 CV**）
   - `search_used`（キーワード検索が使われた回数）
   - `share_clicked`（OGP シェアボタンが押された回数。実装する場合）

1.2. 主要 KPI を定義し週次でレビューする（5 指標以下に絞る）:
   - 週次アクティブユーザー数 (WAU)
   - 平均セッション時間
   - 自治体公式サイトへの外部遷移率（**最重要・代理指標**）
   - 動物詳細ページの閲覧数
   - 流入経路の内訳 (Organic / Direct / Social / Referral)

1.3. Search Console で以下が確認できる:
   - sitemap.xml が `成功` ステータスになっている（1-2 日後に自動）
   - 主要ページ（トップ、47 都道府県 LP）がインデックスされている
   - 検索パフォーマンス画面に impression が出始めている

### 要件 2: SNS の公式アカウント開設と継続運用基盤

**Why**: 検索流入は遅効性。早期に発見してもらうには既存ユーザーを抱えるプラットフォームに乗る必要がある。

#### 受け入れ基準

2.1. **X (Twitter)** で oneco 公式アカウントを開設する。プロフィール文・ヘッダー画像・ピン留めツイートを整える。

2.2. **Bluesky** に同名アカウントを開設する（X のリスクヘッジ＋オープンプロトコル親和性）。

2.3. **Instagram** は判断保留（動物写真の著作権・自治体からの利用許諾が必要なため初期はスキップ可）。

2.4. **AI 自動投稿パイプライン**を設計する:
   - 新着動物情報を日次でピックアップ（DB の `shelter_date` 降順 / `image_urls` あり）
   - Claude で投稿文を生成（テンプレート: 種別、性別、地域、特徴、自治体URL）
   - 画像著作権の観点で、**サムネイル画像はテキストカードで代替**（実画像は自治体サイトへ誘導）
   - 投稿頻度: 1日 1-3 件（過剰投稿でフォロワー疲れを起こさない）
   - 失敗・成果のログを GA4 referral で追跡

2.5. ハッシュタグ戦略: `#保護犬 #保護猫 #里親募集 #迷い犬 #迷い猫 #<自治体名>` を組み合わせる。

### 要件 3: コンテンツ SEO による中長期流入

**Why**: SNS は瞬間風速。**「保護犬 譲渡 <都道府県名>」「迷い猫 探す <市区町村>」**等のクエリで上位を取れば、毎日 24h 流入し続ける。

#### 受け入れ基準

3.1. **47 都道府県 LP (`/areas/[prefecture]`)** が各都道府県名で検索上位を狙えるよう、以下を満たす:
   - 都道府県名 + 「保護動物 / 保護犬 / 保護猫 / 里親 / 譲渡」を title / h1 / description に含む
   - 「現在 N 頭の保護動物が掲載されています」のような動的な数値を表示（最新性アピール）
   - 自治体公式リンクへの導線を明示（信頼性アピール）

3.2. **動物詳細ページ (`/animals/[id]`)** が検索結果に出ても自然な形になる:
   - title: `<種別>の<仮名/管理番号> - <地域>の<カテゴリ>`
   - description: 動物の特徴 + 「<自治体名>公式情報」
   - JSON-LD (Schema.org の `Pet` or `LostFound`) は既に実装済み（[[project_release_audit_2026_06_11]]）

3.3. **任意の追加コンテンツ** (優先度: 中):
   - `/about` の充実（活動内容・運営者情報・問い合わせ）
   - `/blog` or `/news`（譲渡成立事例、自治体取材記事。**ただし無断転載リスクに注意**）

### 要件 4: 自治体・保護団体ネットワークへの露出

**Why**: 自治体・保護団体は里親候補の最初の窓口を持っている。oneco の信頼性を彼らが認めれば、**「ここで一覧で見られますよ」**という形で間接的に紹介してもらえる。

#### 受け入れ基準

4.1. 「oneco を紹介してもらえるテンプレートメール」を用意する:
   - 自治体担当者向け（敬語・公的文体）
   - 保護団体向け（NPO・ボランティア向け）
   - **無断リンク許可申請ではなく、データの再配信について事前告知＋連絡先公開を主旨とする**（適法性確保）

4.2. 「掲載自治体一覧と oneco の運営方針」を公開する `/transparency` or `/sources` 的なページを作る:
   - 全 91 ホストの一覧と公式リンク
   - 収集頻度・更新タイミング
   - 「情報の正確性は最終的に各自治体公式情報に依拠してください」の明記
   - 著作権・撤去依頼の窓口（GitHub Issues + メール）

4.3. 動物保護関連の **Linktr.ee / リストサイト**（既存の里親募集ポータル集約サイト）への自己掲載申請を行う。

### 要件 5: 譲渡成立・再会成立の「成果」を可視化

**Why**: 「このサイトのおかげで○○ちゃんに会えた」という証拠は最強の信頼感醸成。ユーザーが拡散したくなる物語を作る。

#### 受け入れ基準

5.1. アーカイブ機能（[[project_release_audit_2026_06_11]] PR #180）を活用し、`/stats` または `/reunion` で以下を公開:
   - 累計の譲渡成立数（自治体ページが「終了」になった件数の代理指標）
   - 迷子の再会成立数（lost → 削除 = 解決と推定）
   - 都道府県別の活動状況

5.2. ユーザーから「うちの子が見つかりました」「里親になりました」の報告を受け付けるフォーム（GitHub Issues or Google Forms）を用意する（任意・優先度低）。

### 要件 6: ドメイン取得タイミングの判断基準

**Why**: 独自ドメイン取得は集客効率に直結するが、初期のサンクコストとして避けてきた経緯がある。明確な判断基準を残し、後で迷わない。

#### 受け入れ基準

6.1. 以下のいずれかを満たしたら独自ドメインを取得する:
   - **月間 PV 5,000 以上** に到達した時点で `.com` を取得（Cloudflare Registrar、年 1,500円）
   - 自治体・プレスからの問い合わせが**1件でも来た**時点で `.com` を取得
   - 6.1.a/b より早く **マネタイズ（寄付集めページ・クラファン等）を始める**と決めたら、その時点で `.jp` も取得

6.2. ドメイン取得時の作業:
   - Vercel に追加 → DNS 設定 → SSL 自動発行
   - `NEXT_PUBLIC_SITE_URL` を新ドメインに更新（vercel env）
   - Cloud Run `CORS_ORIGINS` に新ドメインを追加
   - Search Console / GA4 に新ドメインのプロパティを追加
   - 旧 `vercel.app` → 新ドメインの 301 リダイレクト設定（過去リンク保護）

## 3. 非目標

- **Google 広告 / Meta 広告等の有料集客**: 初期は試さない。費用対効果の検証ができる規模になるまでオーガニック施策に集中
- **アプリ版開発**: Web で十分。スマホ最適化はレスポンシブで対応済み
- **ユーザー登録・ログイン機能**: Web のみで価値が出せる構造（[[project_vision]] 参照）
- **マネタイズ機能の即時実装**: 弁護士相談前に営利化は進めない（[[project_monetization_legal_checkpoint]]）
- **動物写真の自前ホスティング**: 著作権リスクを避けるため、サムネイルは自治体公式画像へのリンクで代替（既存 next/image remotePatterns で対応済み）
