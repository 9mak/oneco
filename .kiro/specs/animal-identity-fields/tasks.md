# Implementation Plan

> 各 Major タスク = 1 スライス = 1 PR。Slice 0 → 1 → 2 → 3 の順に実装・レビュー（段階レビュー方針）。
> TDD（Red→Green）で進め、各スライスで既存スイートを Green に保つ。Slice 間はコード配線のみ（migration は Slice 0 に集約）。
> 実装ブランチは PR #157 マージ後の main から rebase してから着手する（normalizer/repository/routes が #157 と重なるため）。

- [x] 1. Slice 0 — 個体識別4フィールドの空配線（一気通貫の基盤）
- [x] 1.1 永続化スキーマに4フィールドを後方互換で追加
  - 保護動物テーブルに品種・仮名・性格特徴・管理番号の4属性を、すべて任意（NULL 可）として追加する
  - 品種は検索で使うため索引を付与、性格特徴は長文を許容する型にする
  - 既存行・既存サイトが未充填でも整合性を保つ追加専用のスキーマ変更とし、本番への適用は1回にまとめる
  - スキーマ検証テスト（列の存在・任意性）を追加し、既存データがエラーなく保持されることを確認する
  - _Requirements: 1.1, 1.4, 6.1_

- [x] 1.2 ドメインモデルに4フィールドを任意項目として追加
  - 収集生データと正規化済みモデルの双方に4フィールドを既定値（生データは空文字、正規化済みは未設定）で追加する

  - フィールド省略時に未設定として扱われることを検証するテストを追加する
  - _Requirements: 1.2, 1.5, 6.2_

- [x] 1.3 リポジトリのマッピングに4フィールドを配線

  - 更新時は既存カテゴリ項目と同様に無条件上書きとし、再収集で自然充填できるようにする（古い値の残留を防ぐ）
  - 値あり/未設定の双方で往復しても値が保持されることを検証するテストを追加する

- [x] 1.4 (P) 公開APIスキーマに4フィールドを任意で公開

  - 既存APIクライアントが新項目を無視できる後方互換な追加に留める
  - 4フィールドが任意で公開されることを検証するテストを追加する
  - _Requirements: 1.3, 4.1, 4.4_

- [x] 1.5 (P) フロントエンドの型に4フィールドを任意追加

  - _Requirements: 5.5, 6.3_

- [x] 1.6 基盤の後方互換を検証
